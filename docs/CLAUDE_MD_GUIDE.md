# CLAUDE.md 작성 원칙 가이드

## 왜 CLAUDE.md가 중요한가

- **매 세션 자동 로드** — Claude Code가 가장 먼저 읽는 파일이자 최고 레버리지 지점
- **모든 하위 작업에 전파** — subagent, skill, plan 모드 등 모든 컨텍스트에 영향
- **짧을수록 강력** — 길어질수록 준수율이 떨어지고 핵심이 묻힘

## 적어야 하는 것

| 유형           | 기준                          | 예시                                  |
| -------------- | ----------------------------- | ------------------------------------- |
| 함정 (Gotcha)  | Claude가 틀리면 결과가 치명적 | Materialized View 사용 금지           |
| 기본 행동 억제 | Claude의 일반 원칙과 상충     | 커밋 전 허락 필수                     |
| 빌드/배포      | 코드에서 추론 불가능한 정보   | TestFlight 배포 순서, Vercel 환경변수 |
| 환경 분기      | 잘못 적용하면 프로덕션 사고   | .env.development vs .env.production   |
|                |                               |                                       |

## 적지 말아야 하는 것

| 유형                  | 이유                            | 대안                     |
| --------------------- | ------------------------------- | ------------------------ |
| Claude가 이미 아는 것 | 토큰 낭비, 노이즈 증가          | 삭제                     |
| 린터/포매터 규칙      | ESLint, Prettier가 처리         | .eslintrc, .prettierrc   |
| 특정 작업 전용 지식   | 대화의 <20%에서만 필요          | `.claude/skills/`로 분리 |
| 예제 코드 블록        | CLAUDE.md 비대화의 주범         | Skill에 상세 패턴 기록   |
| 일반적인 코딩 관행    | TypeScript 사용, 에러 핸들링 등 | 불필요                   |

## 핵심 원칙

### Less is More

- **60줄 이하 권장** (HumanLayer 실무 기준)
- 80줄 넘으면 Claude가 일부를 무시하기 시작
- 추가하기 전에 "이걸 빼면 실제로 무엇이 깨지는가?" 자문

### Living Document

- 이론적 필요가 아닌, **실제로 Claude가 틀린 것**만 추가
- 더 이상 필요 없는 규칙은 즉시 삭제
- 분기마다 리뷰하여 사문화된 규칙 제거

### 강조 표현 활용

- `IMPORTANT`, `YOU MUST`, `NEVER` — 준수율 유의미하게 향상
- 단, 남용하면 효과 감소 — 진짜 치명적인 것에만 사용

## Skills 분리 기준

| 조건                   | 위치              | 이유                       |
| ---------------------- | ----------------- | -------------------------- |
| 매 세션 필요           | CLAUDE.md         | 항상 로드됨                |
| 대화의 <20%에서만 필요 | `.claude/skills/` | 필요할 때만 자동 로드      |
| 500줄 이상             | 여러 Skill로 분할 | Skill도 비대하면 효과 감소 |

**SKILL.md 작성 팁**:

- `description`에 트리거 문구 포함 (예: "프론트엔드 파일 수정 시 사용")
- `user-invocable: false`로 설정하면 slash 커맨드 없이 Claude가 자동 로드

## AI 블라인드 스팟 — 경험적 함정 기록법

### 개념 정의

세 계층으로 구분:

| 용어                  | 의미                                           | 예시                                         |
| --------------------- | ---------------------------------------------- | -------------------------------------------- |
| AI 블라인드 스팟      | Claude가 구조적으로 추론 불가능한 영역         | 네이티브 렌더링 quirk, 디바이스 전용 동작    |
| 경험적 함정 (Empirical Trap) | 실기기/런타임 실험으로만 발견 가능한 함정 | LinearGradient가 ScrollView 안에서 클리핑    |
| 디버깅 포스트모템     | 다회차 디버깅 후 다회차 소거법으로 확정한 근본 원인 기록 | A~O variant 테스트 → 원인 특정            |

### 판별 기준

이 중 **2개 이상** 해당하면 "경험적 함정"으로 기록:

- Claude가 3회 이상 다른 원인을 제시했지만 모두 틀림
- 코드 분석만으로는 원인 특정 불가 (런타임/네이티브/디바이스 레벨)
- 소거법(A/B 테스트, variant 분리 등)으로만 원인 확정
- 같은 함정에 다시 빠질 가능성 높음

### 기록 방법: 한 줄 규칙 + 포스트모템

**1단계 — 한 줄 규칙** (MEMORY.md 또는 CLAUDE.md Gotchas 섹션)

형식: `[플랫폼]: [증상] → [원인] — [해결책]`

예시:
> **RN: LinearGradient가 ScrollView 내부에서 시각적 클리핑 발생 — CTA 버튼에는 View + backgroundColor 사용**

**2단계 — 포스트모템** (필요시, `docs/` 또는 `.claude/skills/`)

| 항목      | 내용                                              |
| --------- | ------------------------------------------------- |
| 증상      | 어떤 현상이 보였는가                              |
| 오진 이력 | Claude가 시도한 잘못된 접근 (padding, SafeArea 등) |
| 소거법    | 어떤 variant/실험으로 원인을 좁혔는가             |
| 근본 원인 | 최종 확정된 원인 (한 문장)                        |
| 해결책    | 적용한 수정 (한 문장)                             |
| 조건      | 언제 이 함정이 발생하는가 (재현 조건)             |

2단계는 **반복 발생 가능성이 높거나, 팀에 공유가 필요한 경우**에만 작성.

### 저장 위치 판단

| 조건                         | 위치                            | 이유             |
| ---------------------------- | ------------------------------- | ---------------- |
| 매 세션 Claude가 알아야 함   | CLAUDE.md 또는 MEMORY.md Gotchas | 항상 로드        |
| 특정 도메인(RN, Electron 등) | `.claude/skills/`               | 필요시 자동 로드 |
| 상세 소거법 과정 기록        | `docs/`                         | 참고용 아카이브  |
