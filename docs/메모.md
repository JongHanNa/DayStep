⏺ React Native 모바일앱 테스트 전략 (현업 기준)                                                                    
                                                                                                                 
  현업에서는 모든 경우의 수를 시나리오화하지 않습니다. 리스크 기반으로 우선순위를 매겨 핵심 경로 위주로 구축합니다.
                                                                                                                   
  ---
  1. 테스트 피라미드 (아래→위)                                                                                     
                
  ┌─────────────┬──────┬──────────────────────────────┬───────────────────────────────┐
  │    레벨     │ 비중 │             도구             │             대상              │
  ├─────────────┼──────┼──────────────────────────────┼───────────────────────────────┤
  │ Unit        │ ~60% │ Jest                         │ 비즈니스 로직, 유틸, 스토어   │
  ├─────────────┼──────┼──────────────────────────────┼───────────────────────────────┤
  │ Component   │ ~20% │ React Native Testing Library │ 개별 컴포넌트 렌더링/인터랙션 │
  ├─────────────┼──────┼──────────────────────────────┼───────────────────────────────┤
  │ Integration │ ~15% │ RNTL + MSW                   │ 화면 단위 (API 모킹)          │
  ├─────────────┼──────┼──────────────────────────────┼───────────────────────────────┤
  │ E2E         │ ~5%  │ Detox / Maestro              │ 핵심 유저 플로우              │
  └─────────────┴──────┴──────────────────────────────┴───────────────────────────────┘

  2. E2E 시나리오 — "모든 경우의 수"가 아닌 "Critical Path"

  현업에서 E2E로 커버하는 건 보통 10~20개 핵심 시나리오:

  ✅ 커버하는 것 (Happy Path + 핵심 Edge)
  - 회원가입 → 온보딩 → 메인 진입
  - 로그인 → 메인 → 핵심 CRUD (생성/수정/삭제)
  - 결제 플로우 (구독/복원)
  - 오프라인 → 온라인 복귀 시 동기화
  - 푸시알림 탭 → 딥링크 이동

  ❌ E2E로 안 하는 것
  - 모든 입력 validation 조합 → Unit 테스트
  - 모든 에러 메시지 분기 → Component 테스트
  - 픽셀 단위 UI 검증 → 스냅샷 테스트

  3. 실무에서 많이 쓰는 도구

  Detox (Wix)
  - RN 공식 추천, Gray-box 테스트
  - 실제 디바이스/시뮬레이터에서 실행
  - CI 연동 (GitHub Actions, Bitrise)
  - 단점: 설정 복잡, 느림

  Maestro (요즘 트렌드)
  - YAML 기반으로 시나리오 작성 → 비개발자도 읽기 쉬움
  - 설정이 Detox 대비 훨씬 간단
  - Maestro Cloud로 CI 연동 가능

  # Maestro 예시: 로그인 → 투두 생성
  appId: com.daystep.app
  ---
  - launchApp
  - tapOn: "Google로 시작하기"
  - waitForAnimationToEnd
  - assertVisible: "오늘의 할 일"
  - tapOn: "새 할 일 추가"
  - inputText: "약 먹기"
  - tapOn: "저장"
  - assertVisible: "약 먹기"

  4. 현실적인 구축 순서

  Phase 1: Unit (스토어, 유틸) — 즉시 ROI 높음
  Phase 2: Maestro E2E (핵심 3~5개 플로우) — 배포 안전망
  Phase 3: Component 테스트 (복잡한 UI) — 리팩터링 안전망
  Phase 4: E2E 확장 (엣지 케이스) — 안정기 이후

  5. 화면별 전수 테스트는 언제?

  QA 팀이 있는 조직에서는 테스트 케이스 문서(TestRail, Notion 등)로 화면별 경우의 수를 관리하지만, 이건 수동 QA
  체크리스트이지 자동화 대상이 아닙니다. 자동화는 항상 비용 대비 효과를 따져서 핵심만 합니다.

❯  Liquid Glass 브릿징 구현을 하면, 어떤것들이 바껴? 현재 하단 네비바도 리퀴드 글라스도 바뀌고, 또 그밖에 요약된           
  버튼은 눌렀을때, 요약된 정보를 표시하는 창이 보여지는 동작도 최신 애플 최신  디자인 시스템에서 제공하는 고급스런       
  동작으로 바뀐다던가 그래?                                                                                                
                                                                                                                           
⏺ 좋은 질문입니다. 명확하게 정리해드릴게요.                                                                                
                                                                                                                           
  브릿징이 "자동으로" 바꿔주는 건 없습니다                                                                                 
   
  브릿징은 각 컴포넌트를 개별적으로 구현해야 합니다. "Liquid Glass 브릿징 켜기" 같은 스위치가 아니에요.                    
                
  ---                                
  구현 가능한 것들 (개별 작업)

  1. 하단 탭바

  SwiftUI의 새 TabView (Liquid Glass 소재)를 브릿징하면:
  - 실제 네이티브 glass 소재 (뒤 콘텐츠가 투과되는 진짜 blur)
  - 탭 전환 시 liquid morphing 애니메이션
  - 스크롤 시 자동 숨김/표시 동작

  2. 버튼 → 확장 패널 동작

  iOS 26에서 glassEffectID를 이용한 Matched Geometry + Glass Morphing이 제공됩니다:
  - 버튼이 패널로 "흘러들어가는(liquid)" 전환
  - 같은 glass 컨테이너가 형태를 바꾸는 느낌
  - 이건 SwiftUI 코드로 짜고 브릿징하면 구현 가능

  3. 시트/모달

  - 네이티브 .presentationDetents + glass 소재
  - 손가락으로 잡아당기는 스프링 물리 애니메이션

  DayStep에서 가장 효과 대비 임팩트가 큰 것은:
